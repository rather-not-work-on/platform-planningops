# Unified Personal Agent Platform (nanoclaw + orchestrator + ralph-loop + codex2message)

## What We're Building
`nanoclaw`를 중심으로 개인 에이전트 플랫폼을 구성한다. 목표는 단일 제품처럼 보이되, 내부는 독립 모듈(메인 에이전트, 스케줄러, 플래너, 실행기, LLM 프로바이더, o11y, 메시징 브리지)로 조립 가능하게 만드는 것이다.

핵심 원칙은 다음과 같다.
- `ralph-loop`는 독립 실행 엔진이며 내부 완료 조건을 스스로 판단한다.
- `orchestrator`는 상위 조정자이며 `run queued/running/blocked/terminal` 같은 표준 외부 상태만 본다.
- LLM/Agent 구현체(Codex, Claude Code, local LLM, 서브에이전트)는 동일 실행 계약(contract)에 맞춰 플러그인처럼 교체한다.
- 특정 프로젝트 종속 없이 재사용 가능한 아키텍처를 우선한다.
- 모듈 간 통합은 직접 내부 의존이 아니라 버전 가능한 계약(API/event schema)으로만 연결한다.
- 각 모듈은 독립 배포 가능한 단위로 유지하고, 현재는 함께 운영하더라도 언제든 분리 가능한 구조를 보장한다.

## Why This Approach
요구사항의 핵심은 “통합 제품”이 아니라 “독립 모듈의 캡슐화된 조립”이다. 따라서 내부 구현 공유보다 계약(interfaces/protocols) 중심 설계가 적합하다.

OpenClaw/nanoclaw의 장점(로컬 코드 컨텍스트, 도구 실행, 루프 기반 에이전트)을 메인 에이전트 커널로 활용하고, Ralph Loop를 실행 특화 모듈로 분리하면 확장성과 교체 가능성을 동시에 확보할 수 있다.
또한 provider/agent 엔진을 계약 기반으로 분리하면 Codex/Claude/local LLM 전환 시 락인을 줄이고, 오케스트레이터는 변경 없이 재사용할 수 있다.

## Approaches Considered

의사결정 업데이트: 초기 비교 이후 물리 토폴로지는 `monday` 메인 레포 + `platform-provider-gateway` + `platform-observability-gateway` 독립 레포(하이브리드 분리)로 확정했다.

### Approach A (Recommended): Federated Monorepo (Logical Multi-repo) + Strict Contracts
하나의 모노레포 안에 다수 모듈을 두되, 릴리즈/런타임은 독립 서비스처럼 운영한다.

Pros:
- 초기 개발 속도가 빠르고 통합 테스트가 쉽다.
- 경계를 코드로 강제할 수 있다(contract tests, package boundaries).
- 필요 시 모듈을 별도 레포로 분리하기 쉽다.
- 단일 워크스페이스 생산성과 독립 모듈 원칙을 동시에 가져갈 수 있다.

Cons:
- 규율이 없으면 물리적 레포 분리보다 독립성이 약해질 수 있다.
- 팀 규모가 커지면 CI 파이프라인이 복잡해질 수 있다.

Best when:
- 빠르게 MVP를 만들고, 이후 점진적으로 분리할 계획일 때.

### Approach B: Multi-repo from Day 1
`agent-kernel`, `ralph-loop`, `orchestrator`, `provider-*`, `codex2message-adapter`, `contracts`를 별도 레포로 시작한다.

Pros:
- 캡슐화와 독립 배포가 가장 강하다.
- 기술 스택 다변화(예: executor는 Rust, orchestrator는 TS)가 쉽다.

Cons:
- 초기에 개발/운영 오버헤드가 크다.
- 로컬 개발 경험이 저하될 수 있다(버전 정합, 릴리즈 관리 부담).

Best when:
- 이미 운영 성숙도가 높고, 각 모듈의 릴리즈 주기가 독립적일 때.

### Approach C: nanoclaw Core + External Runtime Mesh
nanoclaw를 사실상 control-plane처럼 두고, `ralph-loop`, Codex executor, local-LLM executor를 원격 실행 노드(mesh)로 붙인다.

Pros:
- 에이전트 실험(새 executor 추가)이 매우 쉽다.
- 개인 환경(로컬/원격/GPU 노드) 혼합 활용에 유리하다.

Cons:
- 관측/보안/권한 모델 설계 난이도가 높다.
- 초기에 장애 지점이 늘어난다.

Best when:
- 멀티 실행 환경과 고급 실험성을 최우선으로 둘 때.

## Key Decisions
- 메인 방향은 `Approach A`로 시작하되, 계약을 `Approach B/C` 수준으로 엄격히 정의한다.
- 물리 토폴로지는 하이브리드로 확정한다: `monday`(메인 레포) + `platform-provider-gateway`(독립 레포) + `platform-observability-gateway`(독립 레포).
- 계약 소유 모델은 하이브리드로 확정한다: C1~C5 공용 계약은 `platform-contracts` 독립 레포 단일 소스, 각 레포 내부 전용 DTO/event는 로컬 소유.
- `ralph-loop`는 완전 독립 모듈로 유지하고, orchestrator는 호출자 역할만 수행한다.
- 상태 저장은 1단계에서 최소화한다(재시작 시 처음부터 재실행). 외부에는 coarse status만 공개한다.
- Provider/Executor는 Template Method 유사 구조를 채택한다: 고정 루프 단계 + 교체 가능한 단계 구현.
- 에이전트 구조는 `nanoclaw` 단일 메인 에이전트로 시작하되, 서브태스크로 다른 에이전트를 호출할 수 있게 설계해 자연스러운 멀티에이전트 확장을 보장한다.
- 관측(o11y)은 `LangFuse`를 기반으로 1단계부터 logs, traces, run timeline을 모두 수집하도록 설계한다.
- 계약 우선 원칙을 채택한다: `Run Lifecycle Contract`, `Subtask Handoff Contract`, `Observability Event Contract`를 표준 경계로 삼는다.
- Agent-native parity 원칙을 채택한다: 사용자가 할 수 있는 핵심 작업은 에이전트도 도구 조합으로 달성 가능해야 한다.
- 실행 완료는 휴리스틱이 아니라 명시적 완료 신호(complete/result)로 판단한다.
- 신뢰 경계 원칙을 채택한다: 오케스트레이터, 실행기, 프로바이더 어댑터는 권한/비밀키 범위를 분리한다.
- P1 이슈(상태 vocabulary, idempotency, completion evidence)는 구현 시작 전 해결해야 하는 선행 조건으로 취급한다.

## Priority Charter (Round 2)
### Non-Negotiable
- 계약 일관성: 상태 머신/오류코드/필수 필드가 문서 전체에서 단일 정의여야 한다.
- 실행 무결성: idempotency, budget, depth, cancel이 계약에서 강제되어야 한다.
- 완료 검증성: `complete`는 증거 스키마 검증을 통과해야만 인정한다.
- 보안 경계: provider 호출은 최소 권한, 비밀 참조, redaction 규칙을 계약에 포함해야 한다.
- 관측 연속성: LangFuse 장애에서도 추적 체인이 보존되어야 한다.

### Negotiable
- 재시작 복구 및 체크포인트
- 고급 UI/워크플로 자동화
- 멀티에이전트 고도 병렬도
- provider 자동 튜닝/스코어링 고도화

## Issue-Driven Refinement (From Review)
### P1 Blockers
- 상태 머신 용어 불일치 (`started` vs `created`, `partial` 매핑 공백)
- scheduler storm 대응 계약 누락(idempotency/dedupe)
- completion evidence 검증 필드 미정

### P2 Hardening
- cancel ack/latency 의미론
- o11y 전달 보장 의미론
- 정책 소유권(Mission Control vs Execution Engine) 경계 명확화
- provider auth scope/secret reference 계약화

### Decision Rule
- 기능 확장 요청은 P1/P2를 우회하지 않는다.
- 새 기능이 기존 계약을 흔들면 기능보다 계약 수정을 우선한다.
- 문서가 늘어나도 계약 경계가 더 명확해지지 않으면 해당 문서는 제거 대상이다.

## Pre-Implementation Verification Order
1. P1 계약 불일치 해소: 상태 머신, idempotency, evidence schema
2. P2 운영 강건성 보강: cancel/o11y/policy ownership/security scope
3. 시뮬레이션 재실행: 장애/중복/지연/권한 에스컬레이션 경로
4. Gate 재검증: A(계약), B(제어), C(관측), D(경계)
5. 구현 착수 판단: 모든 게이트 통과 시에만 `/workflows-plan`

## Key Questions (For Planning)
- 모듈 계약 버전 전략은 무엇으로 할 것인가? (semver, backward compatibility window, deprecation policy)
- `Subtask Handoff`의 최소 표준 필드는 무엇인가? (objective, budget, guardrails, expected artifact, completion contract)
- 실패 처리 기본정책은 무엇인가? (retry, fallback provider, human escalation 기준)
- provider 교체 시 품질 기준은 무엇인가? (정확도, 비용, 지연시간, 도구 호출 신뢰도)
- 멀티에이전트 확장 시 상한은 무엇인가? (동시 서브태스크 수, 최대 깊이, timeout budget)

## Constraints
- `ralph-loop`는 독립 서비스이며 내부 완료 판단을 외부에 노출하지 않는다.
- `orchestrator`는 coarse run status만 소비한다.
- 초기 범위는 재시작 복구 없이 재실행 모델이다.
- `scheduler`는 플랫폼 내부가 아니라 독립 모듈/외부 이벤트 소스다.
- 에이전트 토폴로지는 `nanoclaw` 단일 메인 + 서브태스크 호출 확장 모델이다.
- o11y는 `LangFuse` 기반 logs + traces + run timeline을 기본으로 한다.
- 모듈 통합은 내부 구현 공유가 아니라 계약(API/event schema) 경계로만 수행한다.

## Expected Challenges
- 계약 경계가 느슨해지며 모듈 간 암묵 결합이 생길 위험
- provider별 응답/도구호출 편차로 인한 실행 일관성 저하
- 멀티에이전트 확장 시 디버깅 복잡도 및 비용 급증
- 독립 모듈 운영에 따른 로컬 개발/테스트 오케스트레이션 부담
- 권한 분리 미흡 시 비밀키/실행 권한 경계가 무너질 위험

## Recommended Structure
- `agent-kernel (nanoclaw core)`: 목표 해석, 계획 골격, 서브태스크 분해, 완료 신호 표준화
- `executor-ralph-loop`: 반복 실행, 내부 완료판정, 실패/재시도 전략
- `orchestrator`: run lifecycle 조정, 상태 집계, 모듈 라우팅
- `scheduler`: 독립 트리거 계층(cron/event/webhook)
- `provider-client-adapter`(`monday` 내부): `platform-provider-gateway` 호출 인터페이스
- `platform-provider-gateway`(독립 레포): codex/claude/local-llm 교체 가능한 통합 런타임
- `messaging-adapter (codex2message 계열)`: inbox/outbox, 사용자 상호작용 채널 추상화
- `observability-client`(`monday` 내부): telemetry emit 인터페이스
- `platform-observability-gateway`(독립 레포): LangFuse sink + 공통 run/span/event 수집/전달
- `platform-contracts`(독립 레포): C1~C5 공용 계약 버전/호환성 리포트 단일 소스
- `repo-local contracts`: 각 레포 내부 최적화 DTO/event(외부 공유 금지, 공용 계약으로 매핑 필수)

## Resolved Questions
- `ralph-loop` 내부 완료 조건은 내부에서만 평가하고 외부에는 결과 상태만 노출한다.
- 재시작 복구는 초기 범위에서 제외하고 단순 재실행 모델로 시작한다.
- 1단계 `scheduler`는 플랫폼 내부 내장 대신 독립적으로 분리한다(외부 cron/event 소스).
- 초기 에이전트 토폴로지는 `nanoclaw` 단일 메인 에이전트 + 필요 시 서브태스크 에이전트 호출로 확정한다(점진적 멀티에이전트 확장).
- 관측(o11y) 최소 범위는 `LangFuse` 기반 logs + traces + run timeline으로 확정한다.
- repo 분리 원칙은 `monday` 메인 레포 + `platform-provider-gateway` 독립 레포 + `platform-observability-gateway` 독립 레포로 확정한다.
- 계약 소유 원칙은 \"공용은 중앙, 내부는 로컬\" 하이브리드 모델로 확정한다.
- 외부 상태 모델은 `queued | running | blocked | terminal`로 확정하고, 종료 의미는 `terminal_outcome`(`succeeded|failed|canceled`)으로 분리한다.
- 계약 스키마 1차 소스는 JSON Schema로 확정하고, 추후 변경은 ADR + compatibility gate 통과 시 허용한다.
